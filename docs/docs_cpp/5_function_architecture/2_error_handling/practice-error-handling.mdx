### noexcept

### RAII and Exception Safety

<PracticeProblem
  question="What are the three levels of exception safety guarantees in C++?"
  options={[
    'No-throw, basic, and strong guarantee',
    'Try-guarantee, catch-guarantee, and throw-guarantee',
    'Safe, safer, and safest guarantee',
    'Compile-time, link-time, and runtime guarantee',
  ]}
  correctAnswer={0}
  explanation="No-throw guarantee: the operation will not throw (e.g. destructors, swap, move constructors). Basic guarantee: if an exception is thrown, no resources leak and objects remain in a valid (but unspecified) state. Strong guarantee: if an exception is thrown, the operation has no effect -- the program state is rolled back to exactly what it was before the call (commit-or-rollback semantics)."
  difficulty="medium"
/>

<PracticeProblem
  question="Why must destructors never throw exceptions in C++?"
  options={[
    'It is merely a style guideline with no technical consequence',
    'If a destructor throws during stack unwinding caused by another exception, std::terminate() is called',
    'The compiler automatically marks destructors as noexcept',
    'Throwing from a destructor causes a compilation error',
  ]}
  correctAnswer={1}
  explanation="Destructors are implicitly noexcept (in C++11 and later). If an exception is already active (stack unwinding in progress) and a destructor throws, the runtime calls std::terminate() because it cannot handle two simultaneous exceptions. This is why destructors should swallow errors or use error codes rather than throwing."
  difficulty="medium"
/>

<PracticeProblem
  question="How does the copy-and-swap idiom provide the strong exception guarantee?"
  options={[
    'By catching all exceptions and retrying the operation',
    'By creating a modified copy, then swapping (noexcept) with the original; if the copy fails, the original is untouched',
    'By preventing any exceptions from being thrown during copy',
    'By using std::move instead of std::copy',
  ]}
  correctAnswer={1}
  explanation="The idiom: (1) create a local copy of the object with the desired modifications. (2) If any step throws, the copy is destroyed (stack unwinding) and the original is unchanged. (3) Once the copy succeeds, swap it with the original using a noexcept swap. If swap throws, the state is still valid (basic guarantee). If swap succeeds, the old object is destroyed, completing the commit-or-rollback."
  difficulty="hard"
/>

<PracticeProblem

  explanation="catch(...) is an ellipsis catch handler that matches any thrown exception, regardless of type. Because it matches everything, placing it before any typed catch block would make those blocks unreachable dead code. ### Stack Unwinding

<PracticeProblem
  question="During stack unwinding after an exception, are objects allocated with new on the heap automatically destroyed?"
  options={[
    'Yes, all objects are destroyed during unwinding',
    'No, only stack-local objects are destroyed; heap objects without smart pointers are leaked',
    'Yes, but only if they were allocated in the throwing function',
    'No, nothing is destroyed during unwinding',
  ]}
  correctAnswer={1}
  explanation="Stack unwinding destroys only automatic (stack-local) objects whose constructors completed. Heap memory allocated with new is not tracked by the unwinding mechanism. If the only pointer to a heap object is a local variable, the pointer itself is destroyed but the heap object leaks (the memory is not freed). This is precisely why smart pointers (unique_ptr, shared_ptr) exist: their destructors call delete, providing RAII cleanup for heap objects during unwinding."
  difficulty="medium"
/>
