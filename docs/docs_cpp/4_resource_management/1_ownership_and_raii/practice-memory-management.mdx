### Stack vs Heap

### Pointers and References

### new and delete

<PracticeProblem
  question="Why should you never use delete[] on memory allocated with new (without brackets), and vice versa?"
  options={[

    'The behaviour is undefined and may corrupt the heap',
    'It will leak exactly one element of the array',
    'It is harmless because the allocator handles both uniformly',
  ]}
  correctAnswer={1}
  explanation="Using mismatched new/delete forms causes undefined behaviour. new calls operator new and constructs one object; delete calls the destructor once. new[] may allocate extra memory to store the array count and constructs multiple objects; delete[] reads that count and destructs all elements. Mismatching them means destructors are called incorrectly and the heap metadata is corrupted."
  difficulty="medium"
/>

### Smart Pointers

<PracticeProblem
  question="What is the primary purpose of std::unique_ptr and can it be copied?"
  options={[

    'Automatic garbage collection via a background thread; it can be copied',

  ]}
  correctAnswer={1}
  explanation="std::unique_ptr models exclusive ownership: exactly one unique_ptr owns the managed object at any time. Its copy constructor and copy assignment are deleted, preventing accidental double ownership. Ownership can be transferred via move semantics (std::move). When the unique_ptr is destroyed, delete is called automatically."
  difficulty="medium"
/>

<PracticeProblem

    'Never; reference counts are stored inline in the shared_ptr itself',
    'Only when make_shared is used; it stores a single strong count',

  correctAnswer={1}
  explanation="A control block is allocated alongside the managed object. It stores: the strong reference count (number of shared_ptr instances), the weak reference count (number of weak_ptr instances), the custom deleter (if any), and the allocator (if provided). When the strong count reaches zero, the object is destroyed. When both counts reach zero, the control block itself is freed."

### RAII

<PracticeProblem
  question="What does RAII stand for and how does it prevent resource leaks?"
  options={[
    'Random Access Instruction Interleave; it interleaves memory accesses',
    'Resource Acquisition Is Initialisation; resources are acquired in constructors and released in destructors',
    'Runtime Allocation Integrity Inspection; it checks allocations at runtime',
    'Reference Aliasing Is Illegal; it prevents aliasing bugs',
  ]}
  correctAnswer={1}
  explanation="RAII binds resource lifetime to object lifetime: a resource (heap memory, file handle, lock, socket) is acquired in a constructor and released in the corresponding destructor. Because stack-local objects are destroyed automatically when a scope exits (even via exceptions), RAII guarantees cleanup without manual try/catch or goto, eliminating leaks."
  difficulty="medium"
/>

<PracticeProblem
  question="How does a destructor behave during stack unwinding caused by an exception?"
  options={[
    'Destructors are skipped to improve unwinding speed',
    'Each fully constructed local object on the stack has its destructor called in reverse order of construction',
    'Only the destructor of the throwing object is called',
    'Destructors are called but may themselves throw, replacing the active exception',
  ]}
  correctAnswer={1}
  explanation="During stack unwinding, the runtime destroys all fully constructed automatic objects in the current scope and all enclosing scopes, in reverse order of their construction. This is a core guarantee of RAII: even if an exception propagates, destructors still run, releasing resources. Destructors should never throw (std::terminate is called if a destructor throws during unwinding)."
  difficulty="medium"
/>

### sizeof and Alignment

<PracticeProblem
  question="What does sizeof(int) return on a typical 64-bit system, and is the result a compile-time constant?"
  options={[
    '8 bytes; it is a runtime value',
    '4 bytes; it is a compile-time constant',
    'It depends on whether int is signed or unsigned',
    '4 bytes; it is a runtime value determined by the OS',
  ]}
  correctAnswer={1}
  explanation="On virtually all modern 64-bit platforms (LP64 or LLP64 data models), sizeof(int) is 4 bytes (32 bits). sizeof is a compile-time operator (except when applied to variable-length arrays in C). It returns size_t, an unsigned integer type. The C++ standard guarantees sizeof(char) == 1 but does not specify the size of int beyond requiring it to be at least 16 bits."
  difficulty="medium"
/>

<PracticeProblem
  question="What is memory alignment and why does the compiler insert padding between struct members?"
  options={[
    'Alignment is the size of the entire struct in bytes; padding is added for aesthetic reasons',
    "Alignment ensures each member starts at an address that is a multiple of its type's alignment requirement, improving hardware access speed",
    'Padding is added to prevent buffer overflow attacks',
    'Alignment is only relevant for heap-allocated objects',
  ]}
  correctAnswer={1}
  explanation="Each type has an alignment requirement (in most cases equal to its size for primitive types: alignof(int) == 4, alignof(double) == 8). The CPU can fetch aligned addresses in a single memory transaction; unaligned accesses may require multiple cycles or fault on some architectures. The compiler inserts padding bytes between struct members and after the last member so each member satisfies its alignment constraint, at the cost of increased struct size."
  difficulty="hard"
/>

### Memory Leaks

<PracticeProblem
  question="Which of the following scenarios most reliably causes a memory leak in C++?"
  options={[
    'Declaring a large stack-allocated array in main()',
    'Allocating with new and losing the last pointer to that memory without deleting it',
    'Passing a large object by value to a function',
    'Returning a reference to a local variable from a function',
  ]}
  correctAnswer={1}
  explanation="A memory leak occurs when heap memory is allocated (via new/malloc) and the program loses all pointers to that allocation without freeing it. The memory remains allocated but unreachable for the rest of the program lifetime. Passing by value copies (no leak). Returning a reference to a local is undefined behaviour (dangling reference), not a leak. Stack memory is always reclaimed."
  difficulty="medium"
/>
