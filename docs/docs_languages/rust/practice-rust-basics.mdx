### Traits and Generics

<PracticeProblem question="What is the key difference between impl Trait and dyn Trait in function signatures?" options={[ 'impl Trait uses static dispatch (monomorphization, zero overhead); dyn Trait uses dynamic dispatch (vtable, runtime cost)', 'impl Trait only works with structs; dyn Trait only works with enums', 'impl Trait is for trait definitions; dyn Trait is for trait implementations', 'There is no difference -- they are interchangeable syntaxes', ]} correctAnswer={0} explanation="impl Trait (e.g., fn process(item: impl Display)) causes the compiler to monomorphize the function for each concrete type, producing specialized machine code. This has zero runtime overhead and enables inlining. dyn Trait (e.g., fn process(item: &dyn Display)) uses a fat pointer with a vtable for dynamic dispatch, adding a small indirection cost but allowing heterogeneous collections. Choose impl Trait for performance, dyn Trait for flexibility." difficulty="medium" />

<PracticeProblem question="When should you use an associated type versus a generic type parameter in a trait?" options={[ 'Use an associated type when there is exactly one type per implementing type; use a generic when callers choose the type', 'Use an associated type for all traits; generics are only for functions', 'Use a generic when there is one type per implementation; use an associated type for multiple types', 'They are functionally identical -- always prefer generics', ]} correctAnswer={0} explanation="Associated types (type Item;) allow each implementing type to choose exactly one concrete type. Iterator has type Item because each iterator yields one specific type. You cannot implement Iterator<T> multiple times for the same struct with different Items. Generic parameters (trait Foo<T>) allow callers to specify the type, enabling multiple implementations. Use generics when the caller decides the type, associated types when the implementor decides." difficulty="medium" />

### Concurrency and Unsafe

<PracticeProblem question="What is the interior mutability pattern and which types enable it in Rust?" options={[ 'Interior mutability allows mutation through shared references using Cell (for Copy types), RefCell (runtime borrow checking), or Mutex (thread-safe). The mutation happens inside the container while the container reference remains immutable.', 'Interior mutability means all fields in a struct are mutable by default', 'Interior mutability uses raw pointers to bypass the borrow checker entirely', 'Interior mutability is achieved by making all struct fields pub mut', ]} correctAnswer={0} explanation="Rust normally forbids mutation through shared references (&T). Interior mutability types bypass this in controlled ways. Cell<T> provides get/set for Copy types with no runtime checks. RefCell<T> provides runtime borrow checking: borrow() returns Ref, borrow_mut() returns RefMut, and panics on violation. Mutex<T> and RwLock<T> provide thread-safe interior mutability using OS-level locking. Atomic types (AtomicUsize, AtomicBool, etc.) provide lock-free interior mutability through CPU atomic instructions. The safety invariants are maintained by the type’s API, not by the borrow checker." difficulty="hard" />

<PracticeProblem question="What is cancellation safety in async Rust and how does tokio::select! relate to it?" options={[ 'Cancellation safety ensures that dropping a future (e.g., via select! branch not chosen) does not leave shared state inconsistent. If a future was partway through updating shared state when dropped, that state may be left inconsistent. A future is cancellation-safe if dropping it at any await point leaves the program in a valid state. Many futures are NOT cancellation-safe by default (e.g., splitting a futures::stream::Stream into two readers). Tokio documents which operations are cancellation-safe. To make select! safe, use patterns like completing operations fully before branching, or using AbortHandle for controlled cancellation." difficulty="hard" />
